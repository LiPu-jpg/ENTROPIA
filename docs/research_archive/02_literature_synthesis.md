## A. 直接方法近邻

| 工作家族 | 核心思想 | ENTROPIA 用途 | 造成我们研究空白的局限性 |
| --- | --- | --- | --- |
| Search-R1 / ReTool / 稀疏 GRPO | 使用最终结果奖励训练工具/搜索智能体 | 纯结果锚点基线 | 稀疏反馈在长轨迹上的信用分配较弱。 |
| IGPO / StepSearch / IG-Search | 使用信息增益作为稠密过程奖励 | Utility 信号候选 | 稠密奖励注入过于均匀，在搜索后期可能不可靠。 |
| AutoTool | 使用熵作为优化/损失侧的信号 | 熵使用方式的对比 | 控制不确定性/探索但不路由奖励信号。 |
| SELAUR | 使用不确定性感知的奖励塑形 | Need/不确定性参考 | 将不确定性直接混入奖励值，未分离效用和可靠性。 |
| TIPS / 势函数塑形 | 使用势函数差值作为稠密奖励 | 候选过程信号 | 需要可靠性检查以避免朝代理指标塑形。 |
| TRACE / 奖励作弊分析 | 检测奖励-成功发散和不安全捷径 | Risk 控制器 | 通常用于诊断或约束，未集成为奖励路由。 |
| WorkForceAgent-R1 风格的发现 | 稠密奖励可能退化为作弊 | 动机 | 展示了失败模式但未解决动态信号选择问题。 |

## B. 信号库来源

旧的 CRAFT-Search 方向最好保留为过程信号库，而非整个方法。

| 信号类型 | 可能的度量方式 | ENTROPIA 角色 |
| --- | --- | --- |
| 信息增益 | 答案似然或证据后验的变化 | Utility 信号 |
| 证据覆盖 | 检索到的证据是否支持缺失的子声明 | Utility 信号 |
| 子目标进展 | 针对分解的推理/工具子任务的进展 | Utility 信号 |
| 格式有效性 | 工具调用和答案是否可解析 | 有效性掩码 / 可靠性特征 |
| 效率 | 工具调用次数、token 成本、重复检索 | 成本或风险特征 |
| 安全/作弊 | 重复、引用操纵、奖励-成功发散 | Risk 门控 |

关键转变是：

```text
CRAFT defines S^k_t.
ENTROPIA learns or estimates g^k_t.
```

## C. 研究空白陈述

现有工作通常回答以下问题之一：

1. 应该使用什么稀疏结果奖励？
2. 应该添加什么稠密过程奖励？
3. 不确定性应如何影响优化？
4. 如何检测奖励作弊？

ENTROPIA 的研究空白在于它们之间的交互：

```text
When should a dense process reward be trusted for this state,
this training stage, and this behavior pattern?
```

## D. 基线映射

| 基线 | 测试内容 |
| --- | --- |
| 纯结果 GRPO | 是否需要稠密监督。 |
| 固定稠密 IG | 信息增益是否能单独解释改进。 |
| 固定 CRAFT 复合奖励 | 手工设计的信号库是否足够。 |
| 动态 CRAFT 课程奖励 | 阶段级加权是否足够。 |
| AutoTool 风格的熵损失 | 熵是否应放在损失中而非奖励路由中。 |
| SELAUR 风格的不确定性奖励 | 不确定性塑形是否足够。 |
| ENTROPIA-v1 熵门控 | 仅 Need 是否足够。 |
| ENTROPIA-v2 完整路由 | Need、Utility、Reliability 和 Risk 是否联合必要。 |
| 随机门控 | 收益是否来自有原则的路由而非随机稀疏化。 |

## E. 引用优先级

必须引用并对比：

1. Search-R1 / ReTool 用于稀疏 RL 搜索/工具训练。
2. IGPO / StepSearch 用于信息增益稠密过程奖励。
3. AutoTool 用于熵侧优化。
4. SELAUR 用于不确定性感知的奖励塑形。
5. TRACE 和奖励作弊工作用于可靠性/风险动机。
6. CRAFT 风格的复合奖励工作用于过程信号库构建。

次要引用：

1. TIPS / 基于势函数的塑形。
2. GiGPO / 基于梯度的隐式奖励。
3. HERO / 层次化 RL。
4. 学习型 PRM / 基于验证器的过程监督。
